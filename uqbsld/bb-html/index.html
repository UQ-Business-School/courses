Blackboard Ultra renders custom HTML in one place: an Add HTML block inside an Ultra Document. That single surface is enough for most of what a course needs, all styled on the UQ brand. Two things to take on board before you start.

Not officially supported

Central eLearning does not support the use of custom HTML in Blackboard. This is a Business School approach. It renders reliably inside Ultra Documents and everything on this page has been tested there, but because it sits outside central support, you own what you build with it. For anything a student absolutely must not miss, keep a plain-text version in the page as well.

Only put public material into general tools

Treat anything you paste into a language model as though someone else could read it. Only put publicly available material into general-purpose tools like ChatGPT or Claude. For anything not already public, course content, student information, or unpublished work, use Copilot, which is the University's supported tool and keeps your data within UQ's protections.

The brand files below are all public, so they are fine to paste anywhere. It is the course material you feed alongside them that the rule is about. One honest note on Copilot: it is the right call on the data question, but it follows a long, detailed brand specification loosely at best, so expect to correct more by hand than you would with a more capable model. Use a more capable tool when your inputs are public and you want the full house style held, and Copilot when the material itself cannot leave UQ's protections.

The three pieces

Three things do the work. The instructions file is a single document you paste into a language model so it writes HTML in this style. The component library is the set of tested building blocks further down this page. And the media host is where images and the shared stylesheet live, on the school server, so a page keeps working after a course is copied to a new semester.

That last point matters more than it looks. Blackboard only re-links files it manages through its own editor. Images referenced from raw HTML are invisible to it, so on a course copy their links would break unless they point somewhere course-independent. Hosting them on the school server, at a fixed address that never mentions the course, is what keeps them working year to year.

Getting good output from a model

With the prompt file pasted in, ask for a specific component and give it real content, and you get back a self-contained HTML fragment ready for an Add HTML block. The model has been told to return a fragment rather than a whole web page, to use only the sanctioned components and UQ colours, and to point images at hosted URLs rather than embedding them. You review it, paste it, and refresh the Ultra page to see it render.

Getting a good result

A few habits make the difference between output you can paste and output you have to argue with:

  • Name the component. "Put this in an accordion" works; "make this look nice" does not.
  • Give it the real content and let it style what you provide. It should not be inventing course material.
  • Ask for one fragment per Ultra Document, not a full web page.
  • If a page uses tabs more than once, tell it to give each set its own unique ids, or the tabs will interfere with each other.
  • Paste, save, then refresh. Ultra renders these blocks a moment late, so a first look before refreshing can mislead you into thinking something is broken.

Messages

Callouts

Four states, each carrying meaning from the UQ palette: information, success, warning, and important. Always pair the colour with an icon and a title so meaning never rests on colour alone.

Information

Neutral, helpful context for students.

Success

Confirmation, such as a completed step.

Warning

Something to be careful about, like a deadline.

Important

A critical message students must not miss.

<div class="uqbs-callout uqbs-callout--warning">
  <svg class="uqbs-callout__icon" viewBox="0 0 24 24" fill="currentColor" aria-hidden="true"><path d="M1 21h22L12 2 1 21zm12-3h-2v-2h2v2zm0-4h-2v-4h2v4z"/></svg>
  <div>
    <p class="uqbs-callout__title">Assignment due Friday</p>
    <p>Submit through the assignment link before 5:00pm.</p>
  </div>
</div>
<!-- swap the modifier for the state you need:
     (none) = info, --success, --warning, --error -->

Emphasis

Statement and quote

A statement lifts a single line into large brand type. A quote sets attributed text apart. Use both sparingly; their effect comes from being rare on a page.

Assessment in this course rewards judgement, not recall.

The point of the case method is to be wrong in a room where it is safe to be wrong.

Course convenor

Disclosure

Accordion

For content students can reveal as needed, like FAQs or optional detail. Built on native details and summary, so it is keyboard operable with no JavaScript. Prefer this over tabs for anything important, since it is the more reliable pattern for screen readers.

What is covered in this module?

Open by default. The chevron rotates on toggle.

How is the assessment weighted?

Closed by default, and operable by keyboard out of the box.

<div class="uqbs-accordion">
  <details open>
    <summary>What is covered in this module?</summary>
    <div class="uqbs-accordion__body"><p>Open by default.</p></div>
  </details>
  <details>
    <summary>How is the assessment weighted?</summary>

  </details>
</div>

Tabs

For parallel content in a small space. This version is CSS-only, built on radio buttons and the :checked selector.

Overview panel. Switching tabs needs no JavaScript.

Two things to watch

A screen reader announces these as radio buttons rather than tabs, so for content a student must not miss, use an accordion instead. And every tab set on a page needs its own unique ids and its own line in the checked rule, or two sets will fight.

<style>#t1:checked ~ #p1, #t2:checked ~ #p2{display:block;}</style>
<div class="uqbs-tabs">
  <input type="radio" name="set1" id="t1" checked><label for="t1">Overview</label>
  <input type="radio" name="set1" id="t2"><label for="t2">Details</label>
  <div class="uqbs-tab-panel" id="p1"><p>Overview panel.</p></div>
  <div class="uqbs-tab-panel" id="p2"><p>Details panel.</p></div>
</div>
<!-- give the next set on the page different ids: t3/p3, t4/p4, and a new rule -->

Authoring by hand

If you are pasting components yourself rather than going through a model, a handful of rules keep things predictable:

  • One HTML block per Ultra Document. Put the stylesheet once at the top of that block, then the components under it.
  • Everything sits inside a single <div class="uqbs"> wrapper so the brand styles apply.
  • Use only the components here and only UQ colours. Restraint reads as brand; a rainbow of coloured boxes does not.
  • Point every image at a hosted URL on the school server, so it survives a course copy.
  • Give repeated tab sets unique ids.
  • Save, then refresh the page before you judge the result.

The instructions file and the full component library file live alongside this page. If a component renders oddly, refresh first, then check you pasted into an Add HTML block inside an Ultra Document rather than a basic item. Anything still stuck, or a component you wish existed, is worth a note to the Business School learning design team.
