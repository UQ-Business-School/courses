Part A: Information Systems Development (ISD) Theory

A1: Introduction to Systems Development

What is Information Systems Development?

Information Systems Development (ISD) refers to the process of creating, maintaining, and improving information systems within organisations. It encompasses all activities from initial planning through to ongoing maintenance of operational systems.

Why is ISD Important for Business?

  • Competitive advantage: Well-developed IS can differentiate a business from competitors
  • Operational efficiency: Automates processes and reduces manual work
  • Better decision-making: Provides accurate, timely information to managers
  • Customer satisfaction: Improves service delivery and customer experience
  • Cost management: Proper development reduces long-term maintenance costs

Key Questions Managers Must Answer

  1. How should managers build a business case for the acquisition and development of a new information system?
  2. What are the core problem-solving steps for developing a new information system?
  3. What are the alternative methods for building information systems?

A2: The Systems Development Life Cycle (SDLC)

Definition

The Systems Development Life Cycle (SDLC) is a conceptual framework that describes the stages involved in an information system development project, from initial feasibility study through to system maintenance. It provides a structured approach to planning, creating, testing, and deploying information systems.

Purpose of the SDLC

  • Provides a framework for planning and controlling IS projects
  • Ensures systematic progression through development stages
  • Facilitates communication between stakeholders
  • Helps manage risk by identifying problems early
  • Creates documentation for future reference and maintenance
Key Understanding: The SDLC is a conceptual model that can be implemented using different methodologies. Waterfall is one methodology for implementing the SDLC; Agile/Scrum is another. The SDLC itself is not a methodology—it's a framework that methodologies follow.

A3: Waterfall Methodology — Detailed Analysis

Overview

The Waterfall methodology is a traditional, sequential approach to systems development where each phase must be completed before the next phase begins. It gets its name because progress flows steadily downward through the phases, like a waterfall.

Key Characteristics

CharacteristicDescription
SequentialPhases are completed one after another in a fixed order
Document-drivenEach phase produces detailed documentation as deliverables
Gate-controlledProgress requires formal approval at each phase boundary
Plan-drivenExtensive upfront planning before development begins
Change-resistantChanges after a phase is complete are costly and difficult

The Five Phases in Detail

Phase 1: Systems Definition

Purpose: Establish what the system should accomplish at a high level and determine whether the project is feasible and worthwhile.

Key Activities:

  • Define System Goals: Establish clear, high-level objectives (e.g., enhance decision-making, align with competitive strategy)
  • Determine System Scope: Set boundaries for what the system will and will not include (e.g., system may include sales processes but exclude HR functionalities)
  • Assess Feasibility: Evaluate technical, economic, and operational viability
    • Technical feasibility: Can we build it with available technology?
    • Economic feasibility: Is it financially worthwhile?
    • Operational feasibility: Will users accept and use it?
  • Form a Team: Assemble a skilled team for development and implementation
  • Plan the Project: Create a detailed project plan covering timelines, resources, and deliverables

Deliverable: Project plan with defined scope and feasibility assessment

Gate: Senior management approval to proceed

Example: A retail company wants a new inventory management system. In the Systems Definition phase, they would define goals (reduce stockouts by 30%), scope (covers warehouse and store inventory, excludes supplier management), assess whether they have budget and technical capability, and create an initial project plan.
Phase 2: Requirements Analysis

Purpose: Gather and document detailed requirements from users and stakeholders to understand exactly what the system must do.

Key Activities:

  • Interview users and stakeholders
  • Observe current business processes
  • Document functional requirements (what the system must do)
  • Document non-functional requirements (performance, security, usability)
  • Create detailed requirements specification document
  • Review requirements with stakeholders for approval

Deliverable: Approved user requirements document

Gate: Stakeholder sign-off on requirements

Critical Point: This is the most critical phase for project success. If initial requirements are incorrect, the resulting system will be flawed, leading to costly corrections later. Studies show that requirements errors are the leading cause of project failure.
Common Misconception: "We can fix requirements later." This is dangerous thinking in Waterfall. Changes to requirements after this phase become exponentially more expensive. A requirement error caught in maintenance can cost 100 times more to fix than one caught during requirements analysis.
Phase 3: Design, Development, and Testing

Purpose: Transform requirements into a working system through design, coding, and rigorous testing.

Key Activities:

  • System Design: Create technical architecture and detailed design specifications
  • Component Design: Design individual modules and interfaces
  • Development: Write code, configure software, build database structures
  • Unit Testing: Test individual components in isolation
  • Integration Testing: Test how components work together
  • System Testing: Test the complete system against requirements
  • User Acceptance Testing (UAT): Users verify the system meets their needs

Deliverable: System design documentation, tested software and hardware

Note: This is typically the most complex and time-consuming phase. It requires coordination between business analysts, designers, developers, and testers.
Phase 4: Implementation

Purpose: Deploy the system into the production environment and transition users to the new system.

Key Activities:

  • Install hardware and software in production environment
  • Migrate data from old systems
  • Train users on the new system
  • Create user documentation and help resources
  • Execute cutover strategy (parallel running, phased rollout, or direct cutover)
  • Provide go-live support

Deliverable: Operational system in production

Implementation Strategies:
  • Direct cutover: Old system switched off, new system switched on (risky but fast)
  • Parallel running: Both systems run simultaneously until new system is proven (safe but expensive)
  • Phased rollout: New system deployed to one department/location at a time (balanced approach)
  • Pilot: New system tested with small user group first (good for validating)
Phase 5: Maintenance

Purpose: Keep the system operational, fix problems, and make improvements throughout the system's lifetime.

Key Activities:

  • Corrective maintenance: Fixing bugs and errors discovered after deployment
  • Adaptive maintenance: Modifying system to accommodate changes in environment (new regulations, new hardware)
  • Perfective maintenance: Enhancing system based on user feedback and new requirements
  • Preventive maintenance: Updates to prevent future problems
Critical Understanding — Often the MOST EXPENSIVE Phase:
  • Maintenance typically consumes 60-80% of total system lifecycle costs
  • This phase lasts as long as the system is in use (potentially many years)
  • Often neglected in initial project planning and budgeting
  • May eventually trigger a new SDLC cycle when the system needs replacement

The Cost-Time Relationship in Waterfall

One of the most important concepts in Waterfall is understanding how the cost of changes increases dramatically as the project progresses.

Phase Where Error DiscoveredRelative Cost to FixExplanation
Systems Definition1xJust update the plan document
Requirements Analysis5xRevise requirements documentation
Design/Development10xRedesign and recode affected components
Implementation20xFix deployed system, retrain users
Maintenance100x+Unpick years of built-up dependencies, retrain, redeploy
Real-World Illustration: Imagine discovering during maintenance that the system should have tracked customer purchase history, but this requirement was missed. To add this now requires: redesigning the database, modifying multiple application modules, migrating years of data, retraining all users, updating all documentation. If caught during requirements analysis, it would simply be added to the specification document.

Strengths of Waterfall

  • Clear structure: Easy to understand and manage with defined phases and milestones
  • Comprehensive documentation: Creates detailed records for future reference
  • Predictable: Timelines and budgets can be estimated upfront (if requirements are stable)
  • Quality gates: Formal reviews ensure quality before proceeding
  • Works well when: Requirements are stable, well-understood, and unlikely to change
  • Regulatory compliance: Documentation-heavy approach suits regulated industries

Limitations of Waterfall

  • Inflexible: Difficult and expensive to accommodate changes once a phase is complete
  • Late visibility: Users don't see working software until late in the project
  • Long delivery time: Months or years between project start and usable system
  • High risk: If requirements are wrong, entire project may fail
  • Assumes predictability: Real-world projects often have changing requirements
  • Front-loaded effort: Requires extensive upfront analysis that may prove wrong

When to Use Waterfall

Waterfall is appropriate when:
  • Requirements are clear, fixed, and well-documented
  • Technology is well-understood and stable
  • Project is relatively short and simple
  • Regulatory requirements mandate extensive documentation
  • Organisation values predictability over flexibility
  • Resources with required skills are available throughout

A4: Agile and Scrum Methodology — Detailed Analysis

Overview

Agile is a philosophy and set of principles for software development that emphasises flexibility, collaboration, customer involvement, and rapid delivery of working software. Scrum is one specific framework for implementing Agile principles.

The Agile Philosophy

Agile emerged from the recognition that traditional Waterfall approaches often failed because:

  • Requirements change over time
  • Users often don't know what they want until they see it
  • Long projects lose relevance by the time they're complete
  • Documentation alone doesn't ensure successful systems

Agile Values (from the Agile Manifesto)

Value MoreOver
Individuals and interactionsProcesses and tools
Working softwareComprehensive documentation
Customer collaborationContract negotiation
Responding to changeFollowing a plan

Scrum Framework

Key Concepts

Sprint: A fixed time period (typically 2-4 weeks) during which a specific set of work must be completed. Each sprint produces a potentially shippable product increment.

Product Backlog: A prioritised list of all features, functions, and requirements for the product. Continuously refined and reprioritised.

Sprint Backlog: The subset of product backlog items selected for the current sprint, plus the team's plan for delivering them.

Daily Standup (Daily Scrum): A brief (15-minute) daily meeting where team members share progress, plans, and obstacles.

Sprint Review: Meeting at the end of each sprint where the team demonstrates completed work to stakeholders and gathers feedback.

Sprint Retrospective: Meeting where the team reflects on the sprint and identifies improvements for the next sprint.

Minimum Viable Product (MVP):
A core Agile concept. The MVP is the simplest version of a product that can be released to users to gather feedback. Rather than building a complete system before release, teams build the minimum necessary features, release them, learn from user feedback, and iterate. The goal is to deliver a potentially shippable product at the end of each sprint.
Scrum Roles
RoleResponsibilities
Product OwnerRepresents the customer/stakeholders, prioritises backlog, defines features, accepts completed work
Scrum MasterFacilitates the process, removes obstacles, coaches the team, protects team from distractions
Development TeamCross-functional team that designs, builds, and tests the product (typically 5-9 people)
The Sprint Cycle
  1. Sprint Planning: Team selects items from product backlog for the sprint
  2. Daily Work: Team works on sprint backlog items, holding daily standups
  3. Sprint Review: Team demonstrates completed work, stakeholders provide feedback
  4. Sprint Retrospective: Team identifies improvements for next sprint
  5. Repeat: Next sprint begins immediately

Benefits of Scrum over Waterfall

Potential Downsides and Challenges of Scrum

Challenge 1: Often Not Implemented Correctly
Many organisations claim to "do Agile" but don't follow the principles. They may adopt some terminology without the underlying practices, leading to poor results.
Challenge 2: Mixing Waterfall and Scrum
When organisations move to Scrum, "product owners" are often reconstituted business analysts who used to write lengthy requirements documents. They may lack the credibility and authority to make quick business decisions that Scrum requires.
Challenge 3: Employee Resistance
Some employees resist Scrum because they prefer individual recognition over team collaboration. As one practitioner noted: "We have pockets of resistance. Some don't like to collaborate. They like to be the stars." Scrum's team focus leaves less room for individual heroics and rewards.
Challenge 4: Burnout Risk
The intensity of sprints—delivering working software every 2-4 weeks—can lead to burnout if not managed carefully. Sustainable pace is an Agile principle, but it's often violated under deadline pressure.
Challenge 5: Less Documentation
While Agile values working software over comprehensive documentation, some documentation is still necessary. Teams sometimes use Agile as an excuse to skip important documentation, causing problems later.

When to Use Scrum

Scrum is appropriate when:
  • Requirements are expected to change or are not fully known
  • Customer/user feedback is valuable and obtainable
  • Speed to market is important
  • The organisation supports collaborative, cross-functional teams
  • Leadership is willing to trust teams with decision-making
  • The problem is complex and benefits from iterative discovery

Part B: Excel Functions

B1: VLOOKUP Function — Complete Guide

What is VLOOKUP?

VLOOKUP stands for "Vertical Lookup." It searches for a value in the first column of a table and returns a corresponding value from another column in the same row. The "V" indicates it searches vertically (down a column).

Syntax

=VLOOKUP(lookup_value, table_array, col_index_num, [range_lookup])

Arguments — Detailed Explanation

1. lookup_value (Required)

The value you want to find in the first column of the table.

  • Can be a cell reference (e.g., A1), a value (e.g., 75), or text (e.g., "Marketing")
  • Text lookups are NOT case-sensitive ("SALES" matches "sales" matches "Sales")
  • If the lookup_value is smaller than the smallest value in the first column (when using approximate match), VLOOKUP returns #N/A
Examples of lookup_value:
  • =VLOOKUP(A1, ...) — looks up whatever value is in cell A1
  • =VLOOKUP(85, ...) — looks up the number 85
  • =VLOOKUP("Smith", ...) — looks up the text "Smith"
2. table_array (Required)

The range of cells containing the data table. This must include:

  • The column to search (must be the FIRST column of the range)
  • The column containing the value to return
  • Any columns in between
Critical Rules for table_array:
  • The lookup column MUST be the FIRST (leftmost) column of the range
  • Use ABSOLUTE references ($) when you plan to copy the formula
  • Can use a Named Range instead of cell references
Correct: =VLOOKUP(A1, $D$1:$F$10, 2, FALSE)
The $ signs lock the table reference so it doesn't change when copied.
Incorrect: =VLOOKUP(A1, D1:F10, 2, FALSE)
Without $ signs, copying this formula down will shift the table range, causing errors.
3. col_index_num (Required)

The column number in the table_array from which to return a value.

  • 1 = first column of the table_array
  • 2 = second column of the table_array
  • And so on...
Common Errors with col_index_num:
  • If col_index_num is less than 1 → #VALUE! error
  • If col_index_num is greater than the number of columns in table_array → #REF! error
Example: If your table_array is D1:F10 (3 columns: D, E, F)
  • col_index_num = 1 returns values from column D
  • col_index_num = 2 returns values from column E
  • col_index_num = 3 returns values from column F
  • col_index_num = 4 would cause #REF! error (only 3 columns exist)
4. range_lookup (Optional but Important)

Specifies whether to find an exact match or an approximate match.

ValueTypeBehaviourData Requirement
FALSE or 0Exact matchFinds only an exact match. Returns #N/A if not found.Data does NOT need to be sorted
TRUE or 1 or omittedApproximate matchFinds exact match OR the largest value less than or equal to lookup_valueData MUST be sorted in ASCENDING order
When to Use Each:
  • Use FALSE (exact match) when: Looking up specific items like product codes, employee IDs, names, categories
  • Use TRUE (approximate match) when: Finding which range a value falls into, like grade boundaries, tax brackets, commission tiers, temperature ranges
Common Mistake: Using TRUE (approximate match) when you need an exact match. If your data isn't sorted ascending, approximate match will return wrong results WITHOUT showing an error. This is a silent failure that's hard to detect.

VLOOKUP Limitations

  • Can only look RIGHT: The lookup column must be the leftmost column; cannot return values from columns to the left
  • Column index is hardcoded: If you insert columns, the col_index_num doesn't automatically adjust
  • Only returns first match: If there are duplicate values in the lookup column, only the first one is found
  • Approximate match requires sorted data: And doesn't warn you if data isn't sorted

B2: Combining IF with VLOOKUP

Why Combine IF and VLOOKUP?

VLOOKUP has limitations when dealing with values outside the table range. Combining IF with VLOOKUP allows you to:

  • Handle edge cases (values below the minimum in the table)
  • Provide custom error messages instead of #N/A
  • Create more robust formulas that work for ALL possible inputs

Pattern

=IF(condition, value_if_true, VLOOKUP(...))

Detailed Example from Course

Problem: The temperature lookup table starts at 0 degrees. If someone enters a negative temperature (e.g., -10), VLOOKUP returns #N/A because there's no matching range.

Solution: Wrap VLOOKUP in an IF function to handle negative temperatures.

Formula: =IF(A13<0,"Below zero",VLOOKUP(A13,$D$7:$F$12,3,TRUE))

How it works:
  1. IF first checks: Is A13 less than 0?
  2. If YES (true): Return "Below zero"
  3. If NO (false): Perform the VLOOKUP as normal
Logic flow:
IF(A13<0, "Below zero", VLOOKUP(...))
↓ ↓ ↓
condition if true if false
Key Principle: This technique ensures formulas work for ALL possible input values, not just those within the lookup table range. Professional Excel formulas should handle edge cases gracefully.

Other Use Cases for IF + VLOOKUP

  • Checking if a cell is empty before looking up
  • Providing different messages for different types of errors
  • Applying different logic based on categories
Handle empty cells:
=IF(A1="","Please enter a value",VLOOKUP(A1,$D$1:$E$10,2,FALSE))

B3: XLOOKUP Function — Complete Guide

What is XLOOKUP?

XLOOKUP is a newer, more powerful lookup function that addresses many limitations of VLOOKUP. The "X" suggests its extended capabilities—it can look in any direction and offers more flexibility.

Syntax

=XLOOKUP(lookup_value, lookup_array, return_array, [if_not_found], [match_mode], [search_mode])

Arguments — Detailed Explanation

1. lookup_value (Required)

The value to search for. Same as VLOOKUP.

2. lookup_array (Required)

The range to search. Unlike VLOOKUP, this is ONLY the column (or row) you're searching—not the entire table.

XLOOKUP advantage: You specify exactly which column to search, giving you more control.
3. return_array (Required)

The range from which to return values. This is separate from the lookup_array.

Key Advantage: Because lookup_array and return_array are separate, XLOOKUP can:
  • Look LEFT (return column can be to the left of lookup column)
  • Return multiple columns at once (specify a multi-column return_array)
4. if_not_found (Optional)

What to return if no match is found. If omitted, returns #N/A.

Examples:
  • =XLOOKUP(A1,B:B,C:C,"Not found") — returns "Not found" instead of #N/A
  • =XLOOKUP(A1,B:B,C:C,0) — returns 0 instead of #N/A
  • =XLOOKUP(A1,B:B,C:C,"") — returns empty string instead of #N/A
5. match_mode (Optional)
ValueMeaningUse Case
0 (default)Exact matchLooking up specific items
-1Exact match or next smallerFinding price tiers, grade boundaries
1Exact match or next largerFinding minimum thresholds
2Wildcard matchPartial text matching with * and ?
6. search_mode (Optional)
ValueMeaningUse Case
1 (default)Search first to lastNormal top-to-bottom search
-1Search last to firstFind most recent entry when there are duplicates
2Binary search (ascending)Faster search on sorted data
-2Binary search (descending)Faster search on reverse-sorted data

XLOOKUP Advantages Over VLOOKUP

Task: Find which school James is studying in.

Data: Names in column B, Schools in column G

Result: Returns "Law department"

Task: Find which school Mars is studying in (but Mars doesn't exist).

Formula: =XLOOKUP("Mars",B:B,G:G,"No match is found")

Result: Returns "No match is found" instead of #N/A

Task: Find the Student ID for Keith (ID is to the LEFT of name).

Example 4: Wildcard Search (Find by First Two Letters)
Task: Find the course name for a student whose name starts with "Da".

Formula: =XLOOKUP("Da*",B:B,F:F,,-2)

Explanation:
  • "Da*" — the * wildcard matches any characters after "Da"
  • The -2 in match_mode enables wildcard matching
Result: Returns "Human behaviour" (Daniel's course)
Example 5: Multiple Return Values (Find Daniel's School AND Course)
Task: Return both the school and course for Daniel in one formula.

Formula: =XLOOKUP("Daniel",B:B,F:G)

Explanation: F:G is a two-column range, so XLOOKUP returns values from both columns.

Result: Returns "Human behaviour" and "Business management department" (spills into two cells)
Example 6: Search Bottom to Top (Find Last Student Born in 1998)
Task: Multiple students were born in 1998. Find the name of the LAST one (bottom-most in the list).

Formula: =XLOOKUP(1998,C:C,B:B,,0,-1)

Explanation:
  • The -1 in search_mode searches from bottom to top
  • This finds James (the second 1998 entry) instead of Jack (the first)

B4: Named Ranges — Complete Guide

What are Named Ranges?

Named ranges allow you to assign a meaningful, descriptive name to a cell or range of cells. Instead of referring to cells by their addresses (like D7:F12), you can use a name like "Temperature_Range_Table".

Benefits of Named Ranges

  • Readability: =VLOOKUP(A1,Temperature_Range_Table,2,TRUE) is clearer than =VLOOKUP(A1,$D$7:$F$12,2,TRUE)
  • Maintainability: If the range changes, update the named range definition once instead of every formula
  • Self-documenting: Names describe what the range contains
  • Error reduction: Harder to make mistakes with meaningful names
  • Professional practice: Industry standard for well-built spreadsheets

How to Create Named Ranges

  1. Select the cells you want to name
  2. Go to the Formulas tab
  3. Click Name Manager
  4. Click New
  5. Enter the name (e.g., "Temperature_Range_Table")
  6. Verify the range is correct in "Refers to"
  7. Click OK
Quick Method: Select cells, then type the name directly into the Name Box (the box showing the cell address, left of the formula bar) and press Enter.

B5: Cell Referencing — Complete Guide

Why Understanding References Matters

When you copy formulas in Excel, cell references change based on their reference type. Understanding this is ESSENTIAL for building formulas that work correctly when copied across multiple cells.

The Three Types of References

1. Relative References (Default): A1

References that change when the formula is copied, adjusting relative to the new location.

Example: In cell C1, the formula =A1+B1
  • Copied to C2 becomes: =A2+B2
  • Copied to D1 becomes: =B1+C1
  • Copied to D2 becomes: =B2+C2
Use when: You want the formula to adjust to each row/column (e.g., summing each row's values)
2. Absolute References: $A$1

References that do NOT change when the formula is copied. The $ sign "locks" the column and/or row.

Example: In cell C1, the formula =$A$1+B1
  • Copied to C2 becomes: =$A$1+B2 (A1 stays locked, B1 becomes B2)
  • Copied to D1 becomes: =$A$1+C1 (A1 stays locked, B1 becomes C1)
Use when: Referring to a fixed value like a tax rate, exchange rate, or lookup table
3. Mixed References: $A1 or A$1

References where either the column OR the row is locked, but not both.

ReferenceColumnRowBehaviour When Copied
$A1Locked (A)RelativeColumn stays A; row changes
A$1RelativeLocked (1)Column changes; row stays 1

Why Absolute References Matter for VLOOKUP

Critical Understanding: The table_array in VLOOKUP should almost ALWAYS use absolute references. If you copy a VLOOKUP formula without absolute references, the table range shifts, causing errors or wrong results.
Correct: =VLOOKUP(A8,$D$7:$F$12,3,TRUE)
When copied from row 8 to row 9, becomes: =VLOOKUP(A9,$D$7:$F$12,3,TRUE)
The lookup value changes (good), but the table stays fixed (also good).
Incorrect: =VLOOKUP(A8,D7:F12,3,TRUE)
When copied from row 8 to row 9, becomes: =VLOOKUP(A9,D8:F13,3,TRUE)
The table range shifted down, now pointing to wrong cells!

Keyboard Shortcut: F4 Key

Press F4 while editing a cell reference to cycle through reference types:

  1. A1 → $A$1 (absolute)
  2. $A$1 → A$1 (mixed: row locked)
  3. A$1 → $A1 (mixed: column locked)
  4. $A1 → A1 (relative)

B6: Common Errors and Troubleshooting

#N/A Error

Meaning: "Not Available" — the lookup value was not found.

Causes:

  • The lookup value doesn't exist in the lookup column
  • Spelling mismatch (check for extra spaces, different cases)
  • Using approximate match (TRUE) when lookup_value is smaller than smallest table value
  • Data type mismatch (looking for number but column contains text, or vice versa)

Solutions:

  • Verify the lookup value exists in the table
  • Use TRIM() to remove extra spaces: =VLOOKUP(TRIM(A1),...)
  • Use IF wrapper for values below minimum: =IF(A1<min,"Too low",VLOOKUP(...))
  • Use XLOOKUP with if_not_found parameter
  • Wrap in IFERROR: =IFERROR(VLOOKUP(...),"Not found")

#REF! Error

Meaning: Invalid cell reference.

Causes in VLOOKUP:

  • col_index_num is greater than the number of columns in table_array
  • A column or row referenced in the formula has been deleted

Solutions:

  • Check col_index_num doesn't exceed table width
  • If table_array is D1:F10 (3 columns), col_index_num must be 1, 2, or 3
  • Use Undo (Ctrl+Z) if you accidentally deleted needed cells
Example: =VLOOKUP(A1,D1:E10,3,FALSE) causes #REF!
Why: D1:E10 has only 2 columns, but col_index_num is 3.
Fix: Change to col_index_num=2, or expand range to include a third column.

#VALUE! Error

Meaning: Wrong type of argument or operand.

Causes in VLOOKUP:

  • col_index_num is less than 1
  • col_index_num is text instead of a number
  • lookup_value is more than 255 characters

Solutions:

  • Ensure col_index_num is a positive integer ≥1
  • Check for formula errors in col_index_num argument

Wrong Results (No Error Shown)

This is the most dangerous situation — Excel returns a value, but it's incorrect.

Causes:

  • Using TRUE (approximate match) with unsorted data
  • Using TRUE when you needed FALSE (exact match)
  • Relative references shifted the table_array when formula was copied
  • Looking up in wrong column

Prevention:

  • Always explicitly specify TRUE or FALSE (don't rely on default)
  • If using TRUE, verify data is sorted ascending
  • Always use absolute references for table_array
  • Test formulas with known values before relying on them

Troubleshooting Checklist

  1. ✓ Is the lookup value in the first column of table_array?
  2. ✓ Is col_index_num within the table width?
  3. ✓ For approximate match: Is data sorted ascending?
  4. ✓ For exact match: Does the value actually exist?
  5. ✓ Are you using absolute references for table_array?
  6. ✓ Are there extra spaces in the lookup value?
  7. ✓ Are data types consistent (text vs. numbers)?
